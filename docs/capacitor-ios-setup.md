# Teampurex iOS App — Capacitor + Codemagic Setup

This document is the runbook for shipping the Teampurex iOS app to
the App Store **from Windows**. Yes, that's possible — but iOS
*builds* still require macOS somewhere in the chain. We use
Codemagic (a cloud CI service for mobile apps) so the macOS runner
lives in their data centre, not on your desk.

If you have a Mac or you're OK borrowing one for occasional builds,
skip to the "Local Mac path" section near the end.

---

## What you'll need to set up (one-time)

### 1. Apple Developer Program — **$99 / year**

Sign up at https://developer.apple.com/programs/enroll.

- Use **info.teampurex@gmail.com** (or whichever email you control)
- Entity type: **Individual** (faster approval — 24-48 hours)
  - Organization is also fine but requires D-U-N-S number + verifies for 1-3 weeks
- Two-factor authentication on the Apple ID is **mandatory** —
  enable it before starting enrollment

Approval email arrives → log in to https://developer.apple.com →
your developer team appears in **Membership** section.

### 2. App Store Connect — bundle ID + app listing

Once approved:

1. Go to https://appstoreconnect.apple.com
2. **My Apps → +** (top left) → **New App**
3. Fill in:
   - **Platform**: iOS
   - **Name**: `Teampurex`
   - **Primary language**: English (India)
   - **Bundle ID**: `com.teampurex.app` (matches Android — keep them
     identical for asset-reuse and easier ops)
   - **SKU**: `teampurex-ios-001` (internal tracker — anything you
     want, never shown to users)
   - **User Access**: Full Access

The new app appears in your apps list. Status: "Prepare for
Submission" — that's correct; we add screenshots + metadata later
right before submission.

### 3. Codemagic — cloud macOS builds

Sign up at https://codemagic.io with your GitHub account.

- Free tier: **500 build minutes / month**
- Each iOS build takes ~5-8 minutes → ~60 builds/month free
- Plenty for our shipping rhythm (we build maybe 3-5 times per
  release cycle)

After signup:

1. Codemagic dashboard → **Add application → GitHub**
2. Authorise Codemagic to read your `Purex` repo
3. Select **Purex** from the repo list
4. Choose **Capacitor** as the project type (Codemagic auto-detects
   `capacitor.config.ts`)
5. Click **Next** — Codemagic creates a `codemagic.yaml` config
   template

Configuration we apply to that template is documented in
**section 5** below ("Codemagic build config").

### 4. Generate iOS signing certificates

iOS apps must be signed with a certificate tied to your Apple
Developer account. Codemagic handles this beautifully via
**Automatic Signing** — you upload your Apple credentials once,
Codemagic generates + manages all certs and provisioning profiles
on every build.

In Codemagic:

1. **App settings → Code signing identities → Apple Developer
   Portal**
2. Click **Connect Apple Developer Portal**
3. Provide:
   - Apple ID: your developer-program email
   - App-specific password: generate at https://account.apple.com →
     Sign-In and Security → App-Specific Passwords
   - Team ID: from https://developer.apple.com/account (top-right
     after login, looks like `XXXXXXXXXX`)
4. Codemagic verifies and stores. You never see/touch certs again.

### 5. Codemagic build config

Create `codemagic.yaml` at the repo root (see "What to ship" below
for the exact contents). Once present, every push to `main`
triggers a build → produces a signed `.ipa` → uploads to App Store
Connect → ready for TestFlight.

For now this file doesn't exist yet — we add it the first time we
actually want to build. The setup steps above are what you do
*before* that first build attempt.

## The first iOS build — Codemagic cloud path

Once you have the Apple Developer Program approved + Codemagic
connected to the repo:

### Step 1 — Add `codemagic.yaml` to the repo

```yaml
workflows:
  ios-workflow:
    name: iOS Workflow
    instance_type: mac_mini_m2
    max_build_duration: 60
    environment:
      ios_signing:
        distribution_type: app_store
        bundle_identifier: com.teampurex.app
      vars:
        XCODE_WORKSPACE: "App.xcworkspace"
        XCODE_SCHEME: "App"
        BUNDLE_ID: "com.teampurex.app"
      node: 20.18.0
      xcode: latest
      cocoapods: default
    triggering:
      events:
        - push
      branch_patterns:
        - pattern: 'main'
          include: true
    scripts:
      - name: Install npm dependencies
        script: |
          npm install
      - name: Add iOS platform (first run only)
        script: |
          if [ ! -d "ios" ]; then
            npx cap add ios
          fi
      - name: Sync Capacitor + install pods
        script: |
          npx cap sync ios
          cd ios/App
          pod install
      - name: Set up provisioning profile
        script: |
          xcode-project use-profiles
      - name: Increment build number
        script: |
          cd ios/App
          agvtool new-version -all $(($BUILD_NUMBER + 1))
      - name: Build IPA
        script: |
          cd ios/App
          xcode-project build-ipa \
            --workspace App.xcworkspace \
            --scheme App
    artifacts:
      - ios/App/build/ios/ipa/*.ipa
      - /tmp/xcodebuild_logs/*.log
    publishing:
      app_store_connect:
        auth: integration
        submit_to_testflight: true
        beta_groups:
          - Internal Testing
```

Commit this file. Push to main.

### Step 2 — Watch Codemagic build

Codemagic dashboard auto-detects the push, starts the build. Watch
the live log:
- **Install npm dependencies** (~1 min)
- **Add iOS platform** (~30s — first build only, after this the
  ios/ folder exists in your repo if you commit it, or regenerates
  each build if not)
- **Sync Capacitor + pods** (~2 min)
- **Set up provisioning profile** (auto via Codemagic, ~10s)
- **Build IPA** (~3-5 min — actual Xcode compilation)
- **Publishing to TestFlight** (~30s upload)

Total: ~6-9 minutes for the first build.

### Step 3 — Install on your phone via TestFlight

1. Codemagic uploaded the IPA to App Store Connect → TestFlight
2. Apple processes the build (~10-30 min)
3. You get an email "Build available for testing"
4. Install the **TestFlight** app on your iPhone (App Store search)
5. Open TestFlight → see "Teampurex" → tap **Install**
6. App appears on home screen with the Teampurex icon

TestFlight builds expire after 90 days but you can install
unlimited times during that window. Internal testing (you) doesn't
require Apple review — only public TestFlight beta + production
App Store submission do.

## Local Mac path (if you have / borrow one)

If you have access to a Mac for an hour:

```bash
# On the Mac, in the repo root:
git clone https://github.com/infoteampurex-creator/Purex.git
cd Purex
git checkout main
npm install

# Generate the iOS project (one-time)
npx cap add ios

# Sync config
npx cap sync ios

# Open in Xcode
npx cap open ios
```

Xcode opens with the `App.xcworkspace`. From here:
- **Product → Archive** to build for distribution
- **Distribute App → App Store Connect → Upload**
- Or run on a connected iPhone via the Play button

Commit the generated `ios/` folder back to git so subsequent
Codemagic builds can use it as the starting point (faster than
regenerating each time).

## App Store submission (when you're ready for the public listing)

TestFlight is for you + invited beta testers. The public App Store
listing requires more work:

1. **Screenshots** — at minimum, 1 set for 6.7" display (iPhone 14
   Pro Max sized) showing 3-5 hero screens of the app
2. **App description** — 4000 chars max, sold as the marketing copy
3. **Promotional text** — 170 chars max, shown above description
4. **Keywords** — 100 chars max, helps App Store search
5. **Support URL** — `https://www.teampurex.com/contact`
6. **Privacy Policy URL** — `https://www.teampurex.com/privacy`
   (Apple is strict about this — without it, submission rejects
   automatically)
7. **Category** — `Health & Fitness` primary, `Lifestyle` secondary
8. **Age rating** — questionnaire in App Store Connect, takes ~10 min
9. **Apple's data-use disclosure** — 18-question form about what
   data the app collects (steps, sleep, etc.) and what you do with
   it

Submit for review. Apple's first review of a new app: **24-72 hours
typically**, can be a week if it's borderline (anything fitness/
health related gets extra scrutiny — they look for medical claims).

Common reasons Apple rejects fitness apps:
- Unrealistic transformation promises (we already addressed this
  with the Future Clone "Projected based on consistency" disclaimer)
- Subscription unclear or missing restore-purchase button
- Camera/HealthKit/Push usage strings unclear in `Info.plist`
- Crashes during reviewer testing

After approval: app appears in App Store, indexes within 24h, your
PR redirect-page on www.teampurex.com can deep-link to it.

## What's NOT in v1

- **Apple HealthKit integration** — equivalent of Phase 5's Android
  Health Connect on iOS. Same plugin pattern, but needs separate
  permission scopes + uses different APIs. Defer until after Phase 5
  ships on Android.
- **iCloud sync** — not needed; Supabase backend handles cross-
  device sync via auth cookies.
- **iOS-specific push (APNs vs FCM)** — Capacitor's push plugin
  handles both transparently when set up. Comes with Phase 6.
- **Apple Sign-In** — required if you ship social-only auth, but
  we use email+password so it's optional. Add later if we add
  Google/Facebook signin.

## Troubleshooting

| Symptom | Cause | Fix |
|---|---|---|
| Codemagic build fails with "Bundle ID mismatch" | App Store Connect bundle ID differs from capacitor.config.ts | Make sure both are `com.teampurex.app` exactly |
| TestFlight shows "Processing" forever (>1h) | Build invalid in some subtle way | Check Codemagic logs for warnings; common cause is missing app icon at certain sizes |
| "Invalid Provisioning Profile" | Codemagic's Apple connection expired (~6 months) | Re-connect at App settings → Code signing identities |
| Build succeeds but app crashes on launch | Capacitor config issue, usually `server.url` not reachable | Confirm `https://www.teampurex.com` loads on a real iPhone browser |
| Apple rejects: "Guideline 4.0 — Design / Spam" | Apple thinks the app is a webview shell with no native value | Add a brief "About Teampurex" native onboarding screen (3 slides) so the reviewer sees native UI before the WebView. Comes in Phase 7 if needed. |
